По вопросам 1 и 2 (статусы целей и поле score)
Происходит разделение на параметры для сущности Цель:
- параметр: статус          - В работе / Завершенные / Не оцененные (только для фильтров, т.к. в бд это значит null)
- параметр: Оценка цели     - Истинная / Ложная / Не оцененные (упустил этот пункт, добавил его на бэк) (только для фильтров, т.к. в бд это значит null)

Поэтому твое предположение:
```
score = оценка истинности цели (валидация "3 почему")
status = статус работы над целью (в работе / завершена)
```
Верно.

По другим твоим вопросам:
`Что с rejected (ложная)? Это score: "false"?` - это более не используется, но в общем да, `false` - это ложная цель.
`Если цель ещё не оценена, score = null?` - да.

По вопросам из пунта 3:
- `После интеграции с бэком: rawIdeas и goals объединяются в один массив goals_data с бэка?` - да
- `После интеграции с бэком: Разделение происходит по status: "work" для целей в работе?` - да

По пункту 4: `4. Пагинация - размер страницы В API:`:
- поле `page_size` в ответе - это информационное поле
- поле `page_size` в запросе, пользволяет указывать размер страницы пагинации (в пределах разрешенных лимитов)

По твоему вопросу: `Для Goals можно ли изменить page_size через запрос, или он фиксирован?` - `page_size` можно указать только в эндпоинте `'/api/rest/front/app/goals/steps/get/'`, в остальных местах сейчас все фиксировано.

По пункту 5: `5. Шаги без цели` - нет, шаг без цели не возможен.

По пункту 6:
- `После update-запросов нужно ли делать полный refetch данных или достаточно ответа из update?` - это зависит от того как ты будет релазовывать на фронте и мест где это будет использоваться.
К примеру при создание объектов, будет проще получить данные из ответа на обновление, т.к. там сразу придут id созданных объектов, и ты сможешь далее отпралять их если нужно. Поэтому лучше отталкиваться от конкретной ситуации. 

- `Нужно ли поддерживать offline-режим (localStorage как fallback)?` - полагаю localStorage можно использовать, но нужно чтобы он синхронился с id объектов, которые создаются. И реагировал на ошибки на эндпоинтах.
Т.к. использование localStorage это конечно будет быстрее, но важнее чтобы данные которые видит пользователь, были теми же, что находятся на бэке, чтобы потом у пользователя не было расхождений в понимание данных.
